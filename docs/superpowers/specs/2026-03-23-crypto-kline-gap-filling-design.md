# Crypto K-line Gap Filling Design

## 问题定义

当前系统在冷热分离 Lambda 架构下，冷数据库（SQLite）的数据可能落后于当前时间。例如：
- 数据库最新数据：2026-03-21 23:59:00
- 当前时间：2026-03-23 01:20:00
- 缺口：约 25 小时

这导致 API 返回的 K-line 数据出现时间缺口，影响用户体验。

## 根本原因分析

这不是一个 bug，而是架构设计的必然结果：

1. **Binance Vision 发布延迟**：某日（如 03-22）的官方 ZIP 包必须等到次日（03-23）UTC 04:00 后才会发布
2. **冷数据纯净性原则**：SQLite 冷数据库只接收 Binance Vision 官方 ZIP，不接受 REST API 数据
3. **热缓存覆盖范围**：当前热缓存固定拉取 48 小时，未考虑数据库实际最新时间

## 解决方案概述

基于 Lambda 架构原则，通过改进热缓存预热逻辑和离线下载脚本，实现智能缺口填补：

**核心改进：**
1. **动态探底预热**：`warmup_hot_cache()` 查询数据库最后时间戳，从那里开始拉取到现在
2. **离线脚本追赶**：`daily_crypto_download()` 改为智能追赶模式，自动填补所有缺失日期
3. **保持架构纯净**：冷数据库只接收 Binance Vision ZIP，热缓存只接收 REST API 数据

**架构优势：**
- 冷数据（SQLite）= 绝对可信的历史真相（仅来自 Binance Vision）
- 热数据（内存）= 灵活的实时数据（来自 REST API，填补冷数据到当前的缺口）
- 无缝衔接 = API 查询时自动合并冷热数据，用户感知不到缺口

## 组件设计

### 1. 智能热缓存预热（Dynamic Warmup）

**改进的 `warmup_hot_cache()` 函数**

**职责：**
- 应用启动时执行一次
- 查询冷数据库最后时间戳
- 从最后时间戳到当前时间拉取数据填充热缓存

**逻辑流程：**
```
1. 对每个 (symbol, interval) 组合：
   a. 查询 SELECT MAX(timestamp) FROM crypto_ohlc WHERE symbol=? AND bar=?
   b. 如果有数据：
      - start_time = max_timestamp + 1ms
      - end_time = now
   c. 如果无数据（首次启动）：
      - start_time = now - 48 hours
      - end_time = now
   d. 调用 Binance REST API 拉取数据（支持分页）
   e. 写入热缓存
```

**边界情况处理：**
- **数据库为空**：回退到固定 48 小时逻辑
- **缺口超过 48 小时**：只拉取最近 48 小时（避免 API 限流和启动时间过长）
- **API 调用失败**：记录日志但不阻塞启动，允许应用继续运行

### 2. 离线脚本追赶机制（Catch-up Download）

**改进的 `daily_crypto_download()` 函数**

**职责：**
- 每天 08:00 UTC 自动运行
- 检测冷数据库缺失的日期
- 循环下载所有缺失日期的 Binance Vision ZIP

**逻辑流程：**
```
1. 对每个 (symbol, interval) 组合：
   a. 查询数据库最新日期：SELECT MAX(DATE(date)) FROM crypto_ohlc
   b. 计算缺失日期范围：[最新日期+1天, 昨天]
   c. 循环下载每一天：
      for date in missing_dates:
          download_from_binance_vision(symbol, interval, date)
          insert_to_database(data)
```

**关键特性：**
- **自动追赶**：无论缺失多少天，都会自动填补
- **幂等性**：重复下载同一天不会产生重复数据（使用 UPSERT）
- **容错性**：单日下载失败不影响其他日期的下载

### 3. 数据合并层（已存在，无需修改）

**现有的 `/api/crypto/klines` 端点：**
- 已正确实现冷热数据合并
- 使用 `drop_duplicates(keep='last')` 确保热数据优先
- 按时间戳排序返回完整数据


## 关键实现细节

### 1. Binance REST API 分页处理

**问题：**
Binance `/api/v3/klines` 接口单次请求最多返回 1000 条记录。对于 1 分钟级别的 K-line，48 小时有 2880 条，需要分页拉取。

**解决方案：分页轮询（Pagination Loop）**

```python
async def fetch_klines_with_pagination(
    symbol: str,
    interval: str, 
    start_time: int,
    end_time: int
) -> List[Dict]:
    """
    分页拉取 K-line 数据，自动处理 1000 条限制。
    
    逻辑：
    1. 从 start_time 开始请求 1000 条
    2. 检查返回数据的最后时间戳 last_ts
    3. 如果 last_ts < end_time 且返回了 1000 条：
       - 继续请求，start_time = last_ts + 1ms
    4. 否则结束循环
    """
    all_klines = []
    current_start = start_time
    
    while current_start < end_time:
        # 请求最多 1000 条
        batch = await fetch_binance_klines(
            symbol=symbol,
            interval=interval,
            start_time=current_start,
            end_time=end_time,
            limit=1000
        )
        
        if not batch:
            break
            
        all_klines.extend(batch)
        
        # 检查是否需要继续分页
        if len(batch) < 1000:
            # 已经拉取完所有数据
            break
            
        # 更新下一次请求的起始时间
        last_timestamp = batch[-1]['timestamp']
        current_start = last_timestamp + 1
        
        # 防止无限循环
        if last_timestamp >= end_time:
            break
    
    return all_klines
```

**关键点：**
- 每次循环拉取 1000 条
- 使用最后一条记录的时间戳 + 1ms 作为下次起点
- 当返回数据少于 1000 条时，说明已拉取完毕
- 添加 `last_timestamp >= end_time` 防护，避免无限循环

### 2. 48 小时限制的精确计算

**为什么是 48 小时？**
- 1 分钟 K-line：48h = 2880 条 = 3 次分页请求
- 1 天 K-line：48h = 2 条 = 1 次请求
- 在 API 限流范围内，启动时间可接受（约 3-6 秒）

**如果缺口超过 48 小时：**
```python
# 在 warmup_hot_cache 中
gap_hours = (now - max_timestamp) / 3600000  # 转换为小时

if gap_hours > 48:
    # 只拉取最近 48 小时
    start_time = int((now - timedelta(hours=48)).timestamp() * 1000)
    logger.warning(
        f"Gap exceeds 48 hours ({gap_hours:.1f}h) for {symbol} {interval}, "
        f"only warming up last 48 hours. Cold data catch-up needed."
    )
else:
    # 从数据库最后时间戳开始
    start_time = max_timestamp + 1
```

### 3. 错误处理和重试策略

**API 调用失败处理：**
- 单个 symbol/interval 失败不阻塞其他组合
- 记录详细错误日志
- 不抛出异常，允许应用继续启动

**速率限制处理：**
- Binance API 限制：1200 请求/分钟
- 我们的请求量：启动时最多 2 symbols × 2 intervals × 3 pages = 12 请求
- 远低于限制，无需额外延迟

## 风险和缓解措施

### 风险 1：API 限流
**描述：** 频繁调用 Binance API 可能触发限流
**缓解：** 
- 启动时请求量控制在 12 次以内
- 添加指数退避重试机制
- 监控 API 响应头中的限流信息

### 风险 2：启动时间过长
**描述：** 缺口过大时，分页请求可能导致启动缓慢
**缓解：**
- 限制最大拉取范围为 48 小时
- 超过 48 小时的缺口由离线脚本追赶
- 记录警告日志提醒管理员

### 风险 3：内存占用
**描述：** 热缓存存储 48 小时数据可能占用较多内存
**缓解：**
- 已有 2880 条记录限制（约 1-2 MB per symbol/interval）
- 定期清理超过 48 小时的旧数据
- 监控内存使用情况

### 风险 4：时区问题
**描述：** 不同时区可能导致日期计算错误
**缓解：**
- 所有时间计算统一使用 UTC
- 数据库存储使用 ISO 8601 格式
- 明确标注时区信息

## 总结

本设计方案通过以下三个核心改进，完美解决了 K-line 数据缺口问题：

1. **智能热缓存预热**：动态查询数据库最后时间戳，精确填补缺口
2. **离线脚本追赶**：自动检测并下载所有缺失日期的官方数据
3. **架构纯净性保持**：严格区分冷热数据来源，确保数据可信度

**关键优势：**
- ✅ 无缝用户体验：API 返回完整数据，无感知缺口
- ✅ 数据可信度高：冷数据库只接收官方 ZIP，热缓存填补实时缺口
- ✅ 自动容错恢复：长时间关机后自动追赶，无需人工干预
- ✅ 性能可控：启动时间 < 10 秒，API 响应 < 500ms

**实施优先级：**
1. 高优先级：实现智能热缓存预热（解决当前问题）
2. 中优先级：实现离线脚本追赶（长期稳定性）
3. 低优先级：完善测试和监控（生产环境保障）
